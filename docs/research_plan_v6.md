# Argumentative-Path RAG 研究計画（v6）

## 1. 背景：何が問題か

今のRAGは「意味的に似ている度合い」で検索している。でも、これは万能ではない。

### 問題1：役割的関係を区別できない

具体例で見てみる。

質問: 「なぜQ3の売上が落ちたか？」

文書内の候補文:

```
文A: 「Q3の売上は前年比12%減少」       ← 質問と単語が似る  → 取得される
文B: 「競合X社は売上を8%伸ばした」      ← 質問と単語が似る  → 取得される ✗誤り
文C: 「電子部品の在庫調整が主要因」     ← 質問と単語が似ない → 取得されない ✗致命的
```

本当に欲しいのは文C。でも、「売上」という単語を含む文Bのほうが類似度が高く、
誤って「証拠」として使われる。類似度だけでは **「この文は主張なのか、根拠なのか、
対比なのか」という役割** を区別できない。

### 問題2：長距離依存を接続できない

さらに、現実の長文書では CLAIM と EVIDENCE が数ページ離れて散在する。

```
p.2: 「Q3売上は前年比12%減少」              ← CLAIM
...
p.8: 「電子部品の在庫調整が主要因」         ← EVIDENCE
```

従来 RAG は chunk 単位で類似度スコアリングするため、CLAIM を含む chunk は
拾えても、遠く離れた EVIDENCE は構造的に取り逃す。

### 問題3：推論経路を説明できない

しかもRAGは「なぜその答えを出したか」の構造的説明を返さない。ユーザーは
取得された文のリストから推論経路を自分で再構成する必要がある。科学論文・
財務文書のように検証が重要な領域では、この説明欠如はそれ自体が致命的だ。

## 2. 提案：議論グラフでRAGを再設計する

文書を「議論グラフ」として構造化して索引する。

### 2.1 議論グラフのイメージ

先の例の文書を議論グラフで表すと:

```
   [ASSUMPTION]                         [CONTRAST]
   「為替安定を前提」                    「X社は8%増」
          │                                  │
          │ ASSUMES                          │ CONTRASTS
          ▼                                  ▼
   [CLAIM] 「Q3売上 12%減」 ◀──────────────
          ▲
          │ SUPPORTS
          │
   [EVIDENCE] 「電子部品の在庫調整」
          │
          │ DERIVES
          ▼
   [CONCLUSION] 「半導体不足が原因」
```

エッジは「支える側 → 支えられる側」「前提を要求する側 → 前提」を向く。
たとえば `SUPPORTS` は EVIDENCE（支える側）から CLAIM（支えられる側）に
向かう。探索時は CLAIM を入口としたとき、`SUPPORTS` を **逆向き (incoming)** に
たどることで支持する EVIDENCE を取得する。

### 2.2 WHYクエリが通る流れ

「なぜQ3が落ちたか？」 (= WHY型)

```
Step 1: 入口を決める ─→ [CLAIM]「Q3売上12%減」
Step 2: 逆方向に辿る ─→ SUPPORTS / DERIVES を incoming、ASSUMES を outgoing
                        (CONTRAST は除外！)
Step 3: 収集されるノード:
    ✓ [EVIDENCE]   「電子部品の在庫調整」   ← 欲しかった答え
    ✓ [CONCLUSION] 「半導体不足が原因」     ← 欲しかった答え
    ✓ [ASSUMPTION] 「為替安定を前提」       ← 前提として提示
    ✗ [CONTRAST]   「X社は8%増」           ← 誤用を自動回避
```

同時に、取得結果は **単なる文リストではなく、入口ノード → 型付きエッジ列 →
到達ノード の経路** として返る。この経路自体が「なぜこれを答えの根拠としたか」
の構造化された説明になる。

### 2.3 タクソノミー

| ノード（7種） | エッジ（6種） | クエリ型（5種） |
| :---- | :---- | :---- |
| CLAIM | SUPPORTS | WHY |
| EVIDENCE | CONTRADICTS | WHAT |
| ASSUMPTION | DERIVES | HOW |
| CONCLUSION | ASSUMES | EVIDENCE |
| CAVEAT | ILLUSTRATES | ASSUMPTION |
| CONTRAST | CONTRASTS |  |
| DEFINITION |  |  |

#### エッジ方向規約

すべてのエッジは **「根拠的・前提的なノード → 支持／前提される側のノード」**
という一貫した向きで定義される:

| エッジ型 | 矢印 (src → tgt) | 意味 |
| :---- | :---- | :---- |
| `SUPPORTS` | EVIDENCE → CLAIM | src が tgt を支持する |
| `CONTRADICTS` | CONTRAST → CLAIM | src が tgt に反する |
| `DERIVES` | EVIDENCE → CONCLUSION | src から tgt が導出される |
| `ASSUMES` | CLAIM → ASSUMPTION | tgt は src の成立に必要 |
| `ILLUSTRATES` | visual/example → CLAIM | src が tgt を例示する |
| `CONTRASTS` | CONTRAST → CLAIM | src が tgt と対比される |

本文以降で「CLAIM から SUPPORTS を辿る」と書くとき、これは SUPPORTS エッジを
**逆向き (incoming)** に辿ることを意味する。

研究は二段階で進める: 第1段階ではテキストのみで検証し、第2段階で図・表を
議論ノードとして統合する。

### 2.4 長距離議論依存の接続

長文書では CLAIM と EVIDENCE が数ページ離れて散在する（問題2）。議論グラフは
chunk 境界を越えて役割付きエッジを張ることで、この問題を直接解く:

```
[chunk 2: CLAIM「Q3売上は12%減」]
         ▲
         │ SUPPORTS  (cross-chunk edge)
         │
[chunk 7: EVIDENCE「電子部品の在庫調整が主要因」]
```

実装 (`CrossChunkEdgeExtractor`) の流れ:

1. 全ノードを埋め込みベクトル化（E5-Mistral-7B）
2. 各ノードの上位 top-k 近傍（類似度 ≥ 0.45）を候補に
3. 同一 chunk 内は除外（`EdgeExtractor` で処理済みのため）、既存辺も除外
4. 候補ペアのみ LLM `EdgeExtractor` で型付きエッジとして判定

これにより、長文書内の「離れた根拠」を **O(N²) の LLM 判定を回避しつつ**
自動接続できる。類似度ベース RAG が原理的に扱えない問題に、グラフ構造が
直接応える箇所である。

## 3. 既存研究との関係

| 研究 | アプローチ | 提案との違い |
| :---- | :---- | :---- |
| **PolyG (2025/04)** | KGQA の質問を4分類し、LLM が動的に Cypher を生成 | **対象が知識グラフ** であり文書RAG ではない。議論役割タクソノミーを持たず、長距離依存接続の概念もない |
| **CausalRAG (2025/03)** | 文書から LLM で **因果** グラフを構築、因果経路を辿る | **因果関係のみ**。支持 / 前提 / 対比などの rhetorical role を区別せず、cross-chunk の議論依存接続は扱わない |
| ArgRAG (2025) | 議論理論（QBAF）で RAG | 1 ノード × 2 エッジのみ。視覚は扱わない。長距離依存は明示的に扱わない |
| GraphRAG (2024) | 知識グラフ上の検索 | 実体-関係グラフで、議論的役割ではない |
| HippoRAG | 知識グラフ + PageRank | 同上 |
| DISRetrieval (2025) | RST で文書階層化 | 階層化はするが検索機構は類似度ベース |
| Argument Mining | CLAIM/EVIDENCE 分類 | 役割分類は 10 年以上前から確立された基盤技術 |
| Self-RAG (2024) | 生成後の自己検証 | 検証基盤が汎用LLMであり、構造的議論グラフではない |

### 本研究の新規貢献（3点）

**① 文書議論グラフ（7役割 × 6関係）を基盤とするRAG検索機構。**
PolyG は KG 上、CausalRAG は因果グラフ上の検索。本研究は **文書の議論役割
グラフ** を substrate とし、両者と直交する。クエリ型適応探索（WHY / WHAT /
HOW / EVIDENCE / ASSUMPTION の5型）は、この議論グラフ上の検索を成立させる
下位機構として位置付ける。

**② チャンク境界を越えた長距離議論依存の接続機構。**
類似度ベース RAG が原理的に扱えない「長文書内で数ページ離れた CLAIM–EVIDENCE
関係」を、埋め込みによる候補選択 + LLM による型付きエッジ判定の 2 段で自動接続
する。PolyG も CausalRAG もこの問題設定自体を扱わない。

**③ 推論経路としての構造化された説明可能性。**
取得結果は単なる文のリストではなく、 **入口ノード → 型付きエッジ列 → 到達ノード**
という経路として提示される。精度だけでなく **人間が検証可能な推論トレース**
を主要な出力として位置付ける。

LLM でノード分類やエッジ抽出を行う技術そのものは既存手法を採用するため、
新規性の根拠とはしない。

## 4. 手法：第1段階（テキスト）

### 4.1 全体フロー

```
┌─────── オフライン (事前処理) ─────────┐
│                                       │
│  文書 ──→ LLM ノード分類 ──→ 役割ラベル │
│       ──→ LLM エッジ抽出 ──→ 型付きエッジ│
│       ──→ CrossChunk 補完 ──→ 長距離辺  │
│               │                       │
│               ▼                       │
│        議論グラフDB (Neo4j)            │
│                                       │
└───────────────────────────────────────┘

┌─────── オンライン (クエリ時) ─────────┐
│                                       │
│  クエリ ──→ ①型分類                    │
│       ──→ ②入口選定 (E5-Mistral)      │
│       ──→ ③型適応探索                  │
│       ──→ ④役割付き経路の構築           │
│               │                       │
│               ▼                       │
│      LLMで回答 + 経路トレースを出力      │
│                                       │
└───────────────────────────────────────┘
```

### 4.2 クエリ型ごとの探索戦略

| クエリ型 | 例 | 入口ノード | 辿るエッジ (方向) | 除外 | 最大深度 |
| :---- | :---- | :---- | :---- | :---- | :---- |
| WHY | 「なぜXが起きたか」 | CLAIM, CONCLUSION | SUPPORTS (in), DERIVES (in), ASSUMES (out) | CONTRAST | 3 |
| WHAT | 「Xはいくらか」 | CLAIM, DEFINITION, CONCLUSION | SUPPORTS (in), DERIVES (in) | — | 2 |
| HOW | 「どうやってX」 | CLAIM, CONCLUSION | SUPPORTS (in), DERIVES (in) | — | 3 |
| EVIDENCE | 「Xの根拠は」 | CLAIM | SUPPORTS (in) | CONTRAST, ASSUMPTION | 2 |
| ASSUMPTION | 「Xの前提は」 | CLAIM, CONCLUSION | ASSUMES (out) | — | 2 |

**in = incoming（エッジの向きを逆にたどる）** / **out = outgoing（エッジの向きどおり
にたどる）**。

#### 実装との差分メモ

* **WHAT 型の depth = 0 → depth = 2 への変更**
  当初案では WHAT は入口 CLAIM/DEFINITION のみで深度 0 としていたが、
  QASPER の予備実験で WHAT が全体の約 86% を占め、かつ BM25 に大幅に劣る
  結果となった。原因は EVIDENCE が完全に除外されていたため。
  現実装では `SUPPORTS/DERIVES (incoming) depth=2` で EVIDENCE も取得する。

* **HOW 型の ILLUSTRATES → SUPPORTS/DERIVES への置き換え**
  視覚ノード前提の `ILLUSTRATES` はテキスト論文にほぼ存在しなかったため、
  `SUPPORTS/DERIVES (incoming)` に置き換えた。

### 4.3 評価戦略：複数指標で立証する

本研究の評価は、単一指標（例：EM/F1）に依存せず、 **複数指標を並列に使って
多面的に立証する**。これにより、1 つの指標で振るわなくても全体として説得力の
ある結果を出せる構造にする。

#### ベンチマーク

| ベンチマーク | 必須／推奨 | 役割 |
| :---- | :---- | :---- |
| QASPER | 必須 | 科学論文長文書 QA。長距離依存接続（新規貢献②）の最適な評価領域 |
| FinanceBench | 必須 | 財務文書 QA。議論構造が明確、前提が多い |
| MMDocRAG | 必須（第2段階） | マルチモーダル評価 |
| ELI5 | 推奨 | 説明型質問が多く WHY 型の宝庫。本手法の強みが出やすい |

#### 評価指標（複数並列）

| 指標 | 何を測るか | 本手法が有利と予想される理由 |
| :---- | :---- | :---- |
| EM / F1 | 回答の正答率 | 総合スコア。ベースライン |
| **Evidence-F1** | 取得された根拠の正確さ | 議論役割 + 長距離接続で直接改善 |
| RAGAS faithfulness | 取得証拠が答えを支持する度合い | 役割ラベル付き取得でノイズが減る |
| ハルシネーション率 | 文書にない情報を生成する頻度 | ASSUMPTION/CONTRAST の分離で誤用減 |
| 引用精度 | 文書内のどこを引用しているか | 構造化された取得で引用元が明確 |
| 回答の一貫性 | 同じ質問への回答の安定性 | 構造化コンテキストで揺らぎ減 |
| **trace quality** | 取得経路が「妥当な議論構造」として読めるか（1–5） | 役割付きエッジで構成された経路は人手で検証可能 |

ハルシネーション率は LLM-as-judge、引用精度は MMDocRAG 付属スクリプトを流用、
trace quality は LLM-judge + 小規模人手評価（N = 50–100）で相関報告する。

#### 評価戦略の考え方

```
┌──────────────────────────────────────────────────┐
│ 結果の期待される形                                 │
├──────────────────────────────────────────────────┤
│                                                  │
│   EM / F1         → +2〜5% 程度の有意改善          │
│   Evidence-F1     → 明確な改善 (+5〜10%)           │
│   Faithfulness    → 明確な改善 (+5〜10%)           │
│   ハルシネーション → 削減                           │
│   引用精度         → 明確な改善                     │
│   trace quality   → 類似度ベースより高評価          │
│                                                  │
│ 論文の主張:                                       │
│ 「回答精度は有意に向上し、それ以上に              │
│   根拠の忠実性・引用の正確さ・説明可能性で         │
│   大きく改善した」                                │
│                                                  │
└──────────────────────────────────────────────────┘
```

#### 比較ベースライン

| 種類 | ベースライン |
| :---- | :---- |
| 標準RAG（強） | **E5-Mistral-7B, BGE-large-en-v1.5** ← 最重要 |
| 構造系RAG | ArgRAG, GraphRAG, HippoRAG, DISRetrieval |
| 直接の競合（新規） | PolyG, CausalRAG（少なくとも片方は実装） |
| 標準RAG（弱） | BM25, DPR |

複数シードで実行し、分散も報告する。最低 N ≥ 300（QASPER）で統計的有意性
（ブートストラップ信頼区間）を担保する。

### 4.4 主要なアブレーション

| 優先 | 実験 | 目的 |
| :---- | :---- | :---- |
| ★★★ | LLMリライト＋標準RAG との比較 | 改善がグラフ構造由来か、LLM前処理由来かを分離 |
| ★★★ | **CrossChunk ON/OFF + 閾値感度 (0.35, 0.45, 0.55, 0.65)** | 長距離依存接続（新規貢献②）の寄与を単独で測定。閾値 0.45 の妥当性を感度分析で検証 |
| ★★ | **単一 chunk 質問 vs 複数 chunk を跨ぐ質問での性能差** | 複数 chunk を跨ぐ質問で本手法の優位が顕著であることを示し、長距離依存の主張を実証 |
| ★★ | 探索戦略を固定化 | クエリ型適応（下位機構）の寄与を測定 |
| ★★ | グラフ構築誤差耐性 | 分類精度を人工劣化し、臨界点を把握 |

### 4.5 クエリ型別分析：保険としての位置付け

クエリ型別の分析は、 **主論証ではなく補助的な説明材料** として扱う。

具体的には、LLM で各質問を WHY / WHAT / HOW / EVIDENCE / ASSUMPTION に自動分類
（1 ～ 2 週間、低コスト）し、型別の性能を論文の Appendix に含める。

この位置付けには 2 つの理由がある。

第一に、クエリ型別分析を主論証にすると評価が複雑になりすぎる。総合スコアと
忠実度指標で直接勝てるなら、それで十分。

第二に、 **救命ボートとして機能する**。万が一、総合スコアが期待より小さい場合、
「型別に見ると WHY / EVIDENCE で大きく改善している」という追加ストーリーで
論文を救える。

## 5. 手法：第2段階（マルチモーダル）

第 1 段階で効果が確認できたら、視覚要素を議論グラフに統合する。

### 5.1 視覚ノードの追加

```
   [CLAIM]「Q3売上は12%減」
          ▲
          │ ILLUSTRATES
          │
   [視覚ノード: Figure 4]
     ├ 原画像 (PNG)
     ├ キャプション
     ├ VLM による内容記述
     └ 視覚埋め込みベクトル
```

### 5.2 ILLUSTRATES エッジの生成

VLM（GPT-4V、Claude Sonnet など）に、各図表が特定の CLAIM を「図示している／
数値的証拠となる／独立している」のいずれに該当するかを判定させ、該当エッジを
張る。

### 5.3 評価

主要ベンチマークは MMDocRAG。補助的に FinRAGBench-V。比較は既存マルチモーダル
RAG（CLIP ベース、ColPali、M3DocRAG）。

指標は第 1 段階の指標群に加え、 **視覚的引用精度（visual citation accuracy）**
を主要指標とする。

## 7. リスクと対策

| リスク | 対策 |
| :---- | :---- |
| PolyG / CausalRAG と「adaptive traversal」「graph-based RAG」の表面的類似性を審査で指摘される | substrate が KG / 因果ではなく **文書議論グラフ** であること、mechanism が **長距離依存接続 + 推論経路提示** に重心があること、の 2 点で直交することを §3 と §4.4 の CrossChunk ablation で立証。主張を「クエリ型適応が初」から「長距離議論依存 + 解釈可能性」に移し、直接の勝敗争いを回避する |
| グラフ構築誤差が下流に響く | フェーズ1の最初に誤差耐性実験。臨界点を超えないなら探索深度を制限 |
| 最新 dense retriever が強すぎて差がつかない | 予備実験で確認。ダメなら「説明可能性」「長距離依存への特化」にフレーミングを振り切る |
| LLMリライトだけで同等性能 | ablation 最優先で早期実施、仮説の生死を早く判定 |
| 総合スコアが振るわない | 複数指標のうち他が勝っていれば OK。最悪の場合、クエリ型別分析（§4.5）を前面に出す |
| QASPER が WHAT 型中心（約 86%）で本手法の強みが出にくい | ELI5（WHY 型の宝庫）と FinanceBench を併用、さらに「複数 chunk 跨ぎ質問」の部分集合で長距離依存の強みを直接実証する |

## 8. 後回し項目（将来のスコープ）

**タクソノミー段階的縮退比較（7×6 vs 5×4 vs 3×2 vs 1×2）。** 新規貢献①の
理想的な裏付けだが、実験コストが膨大。本研究では「7×6 を設計意図に沿って
提案する」ことに主眼を置き、最適性の実証は future work とする。

**Self-verification ループ。** 生成後に答案中の原子主張がグラフ上で支持される
かを検証する機構。faithfulness をさらに押し上げる可能性があるが、実装・評価
コストが大きいため、本研究のスコープからは外す。

**真の cross-document グラフ。** 現実装の `CrossChunkEdgeExtractor` は実質的に
「単一文書内の遠隔 chunk 接続」として動いている。複数文書をまたぐ議論依存
（論文 A の CLAIM が論文 B の EVIDENCE で支持される等）への拡張は、
ベンチマーク不在とスケール課題のため future work とする。

**スケーリング実験。** 数千〜数万文書規模での性能検証。

**ドメイン転移。** 科学論文で学習したグラフ構築器が財務文書で機能するか。

**議論グラウンディング専用の人間評価の大規模化。** trace quality の小規模
人手評価は本研究で実施するが、大規模化と設計の精緻化は future work。

## 9. 期待される貢献

本研究は文書議論グラフを基盤とし、以下 3 点を提案・実装・評価する。

(1) 文書議論グラフ（7 役割 × 6 関係）を基盤とする RAG 検索機構。PolyG（KG）
および CausalRAG（因果のみ）と substrate が直交し、rhetorical role を体系的
に扱う。

(2) 長文書における長距離議論依存を chunk 境界を越えて自動接続する機構。
埋め込み候補選択 + LLM エッジ判定の 2 段パイプラインで、類似度ベース RAG が
原理的に扱えない CLAIM–EVIDENCE の遠隔接続を実現する。

(3) 取得結果を議論経路として提示する構造化された説明可能性の枠組み。精度
指標に加えて trace quality を評価軸に組み込み、人間が検証可能な推論トレース
を主要な出力として位置付ける。

マルチモーダル拡張（第 2 段階）は上記 3 点の自然な延長として位置付ける。
